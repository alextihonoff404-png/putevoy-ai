# Сервис подготовки путевых листов — план MVP

Рабочее название: **Путевой.AI** (или **Travelpoint** / **WayBill.pro**).
Прообраз: соседний проект primarydoc.
Цель MVP: за 4–6 недель закрыть весь сценарий «зашёл → настроил один раз → ежемесячно генерирует» для двух сегментов клиентов (ИП и автопарки), с гибридом шаблонов и LLM-парсингом заправок.

---

## 1. Видение и сегменты

**Один продукт, два тарифа** с общим ядром и переключаемой иерархией владения.

| Сегмент | Tier | Цена/мес | Ограничения |
|---|---|---|---|
| ИП / самозанятые / директор-водитель | **Solo** | 490 ₽ | 1 ТС, 1 водитель, 1 шаблон, до 30 листов/мес |
| Малый и средний автопарк | **Fleet** | 2 900 ₽ | до 10 ТС/водителей, роли, общий каталог адресов |
| Корпоративный | **Enterprise** | от 9 900 ₽ + per-ТС | без лимитов, SSO, API, аудит-лог |

Триал: 14 дней / 1 полный месяц генерации без оплаты. Биллинг: ЮKassa (для RU) + Stripe (для зарубежных карт).

## 2. Технический стек

| Слой | Выбор | Почему |
|---|---|---|
| Frontend | Next.js 14 (App Router) + Tailwind + shadcn/ui | Тот же стек, что primarydoc — переиспользуем компоненты |
| Backend | **Python 3.12 + FastAPI** | Вся работа с .xlsx — это `openpyxl`. Алгоритм генерации = чистый код с golden-тестами. |
| БД | PostgreSQL 16 + Prisma (через Node-API) или SQLAlchemy 2 | Реляционная модель (workspace → vehicles → routes → runs) |
| Очереди | Celery + Redis | Генерация .xlsx и конвертация в PDF — фоновые задачи (10–60 с) |
| Хранилище файлов | S3-совместимое (Yandex Object Storage) | Шаблоны + сгенерированные .xlsx, signed URLs |
| Auth | Clerk (быстрый старт) или собственное на NextAuth | Email + OAuth Google/Yandex |
| Биллинг | ЮKassa (RU) + Stripe (международные) | Подписки, триалы, прокси-чеки |
| PDF-превью | LibreOffice headless (`soffice --convert-to pdf`) | Чтобы пользователь видел, как выглядит лист до печати |
| LLM | Claude Haiku 4.5 (через Anthropic API) | Дёшево (~0.5 коп/парсинг), быстро (< 1 с), хороший русский |
| Хостинг | Yandex Cloud (k8s или Serverless Containers) + CDN | Соответствие 152-ФЗ — данные водителей внутри РФ |
| Observability | Sentry + OpenTelemetry → Grafana Cloud | Ошибки генерации критичны, нужны алерты |

## 3. Модель данных (упрощённая)

```
users
  └─ memberships (role: owner/admin/driver)
       └─ workspaces (план, биллинг)
            ├─ organizations (реквизиты, инженер-механик)
            ├─ vehicles (марка, госномер, объём бака, марка топлива, базовый адрес)
            ├─ drivers (ФИО, СНИЛС, в/у, дата выдачи)
            ├─ routes (адрес, км×1сторону, расход×1сторону, флаг is_large)
            ├─ templates (тип: preset|custom, .xlsx blob ref, cell_map JSON)
            ├─ vehicle_states (vehicle_id, last_odometer, last_fuel_balance, last_date)
            └─ monthly_runs
                 ├─ id, year, month, status, vehicle_id, driver_id, template_id
                 ├─ fuelings[] (date, liters, price_per_l, sum, source: manual|llm|photo)
                 ├─ generated_days[] (date, routes[], odometer_start, fuel_balance_end)
                 ├─ output_waybill_xlsx (s3 ref)
                 ├─ output_fuel_xlsx (s3 ref)
                 └─ validation_report JSON
```

Ключевое: **`vehicle_states` — единственный источник правды о переходе между месяцами**. После успешной генерации обновляется атомарно. Если пользователь правит уже сгенерированный месяц — пересоздаётся state.

## 4. Ядро: модуль `generator/`

Самая ценная часть проекта — детерминированный движок. Структура пакета:

```
generator/
  ├─ calendar.py        # рабочие дни месяца РФ (учёт переносов: либ. workalendar.europe.Russia)
  ├─ route_planner.py   # назначение маршрутов по дням (рандом с seed = year*100+month+vehicle_id)
  ├─ fuel_balance.py    # пересчёт остатка, правило бака 50 л, выбор overflow-маршрутов
  ├─ trip_scheduler.py  # время выезда/прибытия (≤ 4 поездок, 09:00, шаг 3 ч)
  ├─ validators.py      # все 6 проверок из инструкции (BU52/BU53 ≤ tank, FS31=AA43, …)
  ├─ writers/
  │    ├─ waybill_writer.py    # клонирует шаблон, заполняет cell_map
  │    └─ fuel_log_writer.py   # дописывает к историческому Топливо.xlsx
  └─ tests/
       ├─ test_april_2026.py   # golden test против реального файла пользователя
       ├─ test_march_2026.py
       └─ test_february_2026.py
```

**Принципы:**
- Чистые функции на входах-выходах (`MonthlyInput → MonthlyOutput`), без I/O.
- I/O (чтение/запись xlsx) изолировано в `writers/`.
- Все константы (объём бака, время начала, шаг) — в конфиге ТС, не в коде.
- **Golden-тесты** — взять февраль/март/апрель 2026 из реальных файлов, в тесте подать на вход «постоянные данные + заправки», на выходе сравнить с реальным `.xlsx` по числу ячеек. Если кто-то меняет алгоритм — тесты падают.

## 5. Стратегия шаблонов (гибрид)

**Cell map** — JSON-схема, описывающая, какая переменная попадает в какую ячейку. Пример:

```json
{
  "name": "preset_mintrans_390_v1",
  "version": 1,
  "front_side": {
    "day_left": {
      "day": "AD2", "month": "AI2", "year": "AX2",
      "release_dt": "BP33", "return_dt": "AA40",
      "odo_start": "BU31", "odo_end": "AA43",
      "fuel_issued": "BU49", "fuel_balance_start": "BU52", "fuel_balance_end": "BU53"
    },
    "day_right": { "day": "EB2", "month": "EG2", "year": "EV2", "...": "..." }
  },
  "back_side": {
    "day_1": { "from_col": "AK", "to_col": "AP", "km_col": "BC", "dep_col": "AU", "ret_col": "AY", "start_row": 7 },
    "day_2": { "from_col": "F",  "to_col": "K",  "km_col": "X",  "dep_col": "P",  "ret_col": "T",  "start_row": 7 }
  }
}
```

**Пресеты (старт):**
- `preset_mintrans_390_v1` — стандартная двусторонняя форма (как в вашем апреле 2026).
- `preset_mintrans_152_simple_v1` — упрощённая односторонняя для самозанятых.

**Загрузка своего шаблона (премиум-фича Fleet/Enterprise):**
- Пользователь загружает пустую .xlsx-форму.
- Запускается **мастер разметки**: пошагово показываем поля («где день?», «где одометр выезда?»), пользователь кликает по ячейкам → собираем cell_map.
- Сохраняем как `templates.type = 'custom'`, привязываем к workspace.
- Альтернатива (для V2): автодетект через LLM по содержимому ячеек ("Дата выезда", "Одометр") — не для MVP.

## 6. LLM-парсинг заправок

Один эндпоинт: `POST /api/v1/fuelings/parse`.

**Вход:** свободный текст:
```
15.04 47.63л по 69.37 = 3300.79
23 апреля 45,69 ₽69.26 3164.07
```

**Системный промпт** (фиксированный, шаблонизированный):
```
Ты парсер чеков на бензин. Извлеки список заправок из текста.
Верни JSON-массив: [{"date":"YYYY-MM-DD","liters":number,"price_per_l":number,"sum":number}]
Если поле отсутствует — null. sum = liters * price_per_l, если не указано.
Год — текущий, если не указан явно.
```

**Параметры:**
- Модель: `claude-haiku-4-5`
- Кэш на системный промпт (Anthropic prompt caching)
- Стоимость: ~500 токенов вход + 200 выход = ~0.3 коп на парсинг
- Fallback: если LLM ответил невалидным JSON — показываем форму ручного ввода

**Что НЕ делаем через LLM:**
- Алгоритм генерации (детерминированный код)
- Назначение маршрутов
- Расчёт остатков
- Валидацию

## 9. Риски и митигации

| Риск | Митигация |
|---|---|
| Разнообразие шаблонов у разных компаний | Пресеты + custom-шаблоны с визуальным маппингом ячеек |
| Юридические вопросы («помогаем фальсифицировать?») | В оферте — «инструмент для оформления уже совершённых поездок согласно отчётам водителя». Не генерируем поездки в будущем. |
| Сложность ручной правки сгенерированного дня | В MVP — простая «перетасовать с другим seed»; ручное редактирование — V2 |
| Конкуренты (1С Путевые листы, Wialon, СберКорус) | Дифференциация: для микро-бизнеса — на порядок проще и дешевле; для автопарков — современный UX |
| Хранение ПДн водителей (СНИЛС, в/у) | Шифрование at-rest, согласие на обработку, регистрация в Роскомнадзоре оператором ПДн |
| Стоимость LLM при росте | Кэш промпта, batch-обработка списка заправок одним вызовом, лимит 100 заправок/месяц на одного пользователя |

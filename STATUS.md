## 1. Что это и для кого

**Веб-сервис автоматической подготовки путевых листов** (форма Минтранс 390) для ИП и малого автопарка. Пользователь один раз настраивает профиль (организация, водитель, ТС, адреса), каждый месяц вставляет заправки → нажимает «Сгенерировать» → получает готовые xlsx путевых листов + журнал учёта топлива + PDF-превью.

**Главная архитектурная ставка**: алгоритм генерации — детерминированный Python-код, не LLM. Это даёт повторяемость, тесты, дешевизну.

## 2. Что СДЕЛАНО (40+ задач)

### Ядро алгоритма (Python, чистые функции)

- **`calendar.py`** — рабочие дни РФ (workalendar, с праздниками и переносами)
- **`trip_scheduler.py`** — расписание поездок: 09:00, шаг 3 ч, до 4 поездок/день
- **`route_planner.py`** — назначение маршрутов на дни:
  - Сегментация по дням заправок (бюджет считается локально для каждого сегмента)
  - **Сглаживание** расхода: каждый день стремится к среднему `budget / remaining_days`
  - Веса маршрутов из «как часто ездите» (Редко 0.5 / Иногда 1.0 / Часто 2.0 / Каждый рабочий день 5.0)
  - **Штраф за повторение** вчерашнего маршрута — естественное чередование
  - Автоопределение «крупных» маршрутов (≥ 5 л на круг) для овэрфлоу
- **`fuel_balance.py`** — правило бака 50 л + овэрфлоу через крупные маршруты
- **`generate.py`** — оркестратор + автоподбор seed (до 500 попыток)
- **`validators.py`** — 6 проверок инструкции (бак ≤ tank, одометр непрерывен, время ≤ 21:00, и т.д.)
- **Дни простоя** (`MonthlyInput.skip_dates`) — исключение рабочих дней (отпуск/болезнь/командировка/праздники сверх календаря)

### Writers (генерация .xlsx и PDF)

- **`waybill_writer.py`** — клонирует встроенный шаблон под N пар дней, заполняет ячейки по `cell_map`, ставит сквозную нумерацию ПЛ (1, 2, 3, …). Симметричные поля печати — двусторонняя печать совпадает 1-в-1.
- **`fuel_log_builder.py`** — собирает накопительный журнал из истории прогонов в БД с формулами-ссылками между месяцами
- **`pdf_preview.py`** — конвертация xlsx → pdf через LibreOffice headless. Graceful fallback если LO не установлен.

### Встроенные ресурсы

- **`builtin_templates/preset_mintrans_390_v1.xlsx`** — очищенная пара листов, клонируется генератором
- **`cell_maps/preset_mintrans_390_v1.json`** — карта 30+ ячеек переменных полей

### Storage (SQLite + SQLAlchemy 2.0)

- **Таблицы**: `profile`, `driver`, `vehicle`, `vehicle_state`, `route`, `monthly_run`, `fueling_record`, `generated_day`
- **Multi-vehicle**: `Vehicle` множественный, `Route`/`VehicleState`/`MonthlyRun` привязаны к `vehicle_id`, в `Profile` — `active_vehicle_id`
- **Идемпотентная перегенерация**: повторный `save_run` использует тот же carry_over из day[0]
- **Удаление прогона** с корректным откатом VehicleState
- **Удаление ТС** с каскадным удалением маршрутов/прогонов/состояния (последнее ТС удалить нельзя)
- **Авто-миграция БД** через `ALTER TABLE ADD COLUMN` для апгрейда старых схем

### Парсинг и геокодинг

- **Regex-парсер заправок** из свободного текста (6 форматов)
- **Импорт стартового состояния** из старого Топливо.xlsx (последняя строка → одометр + остаток + дата)
- **Парсер существующего адреса** (`address_parser.py`) — разбирает строку типа «Санкт-Петербург, ул. Репищева, д. 10, к3» на компоненты (город / тип / название / дом / корпус), и обратно собирает в формат для геокодера
- **Геокодинг**:
  - Яндекс Геокодер (если задан `YANDEX_GEOCODER_KEY`) + fallback на Nominatim
  - OSRM для расстояния по дорогам
  - **3 попытки** с возрастающей агрессивностью: оригинал → раскрытие сокращений → компактный формат (`Репищева 10к3`)
  - Раскрытие 14 сокращений: «пр.» → «проспект», «ул.» → «улица», «д.» → «дом», «к.» → «корпус» и т.д.

### Веб-приложение (FastAPI + Jinja2 + HTMX + Tailwind CDN)

**6 экранов** в современном SaaS-стиле:

- **`/setup`** — мастер из 4 секций (Организация / Водитель / ТС / Стартовое состояние)
  - Структурированный ввод базового адреса (город / тип-селектор / название / дом / корпус) + live-превью
  - Заметный блок «Загрузить журнал учёта топлива за предыдущий период» с пунктирной рамкой
- **`/vehicles`** — список ТС с inline-редактированием, добавлением, удалением, переключением активного. Красная точка-маркер активного ТС.
- **`/routes`** — каталог адресов с CRUD. **Структурированный ввод** адреса (тип-селектор / название / дом / корпус). Кнопка 🔍 — геокодинг от базового адреса.
- **`/dashboard`** (видно как «Обзор») — 4 stat-карточки (одометр / остаток с прогресс-баром / месяцев в истории / следующий месяц), таблица прогонов с action-chips
- **`/generate`** — три секции: «Какой месяц» / «Заправки месяца» / «Дни простоя». Live-пересчёт прогноза. При смене месяца — перезагрузка с новыми данными.
- **`/history`** — все прогоны + ссылки на скачивание + кнопка «полный журнал»

Действия в таблицах оформлены как **action-chips** (нейтральный / primary / success / danger) с цветовым кодированием.

### PDF превью

- Endpoint `/preview/waybill/{year}/{month}` конвертирует xlsx → pdf через LibreOffice
- RFC 5987-кодирование русских имён файлов (Content-Disposition)
- Graceful fallback: HTTP 503 с понятным сообщением и ссылкой на установку LO

## 4. Ключевые архитектурные решения (история «почему»)

1. **Алгоритм в коде, не в LLM** — повторяемость, тесты, дешевизна
2. **Шаблон путевого листа клонируется** — официальная форма с 188 столбцами и сотнями merged cells не пересоздаётся, а клонируется и заполняется по карте ячеек
3. **Автоподбор seed** — пользователь не думает о «магических числах»
4. **Сегментный планировщик** — бюджет распределяется в рамках периода между заправками, не на весь месяц
5. **Сглаживание расхода + штраф повторений** — даёт реалистичное распределение маршрутов (как в живых данных пользователя), без 10 одинаковых дней подряд
6. **Idempotency перегенерации** — повторная генерация того же месяца использует carry_over из day[0] существующего прогона
7. **Multi-vehicle на уровне модели**: `current_vehicle_id()` в repo возвращает активное; все queries фильтруют по нему. Структурно готово к мультитенантности, останется добавить `user_id` к таблицам
8. **`_to_float` хелпер** — поля десятичных чисел принимают и точку, и запятую
9. **CSS-переменные** для палитры — позволят легко добавить тёмную тему позже
10. **Структурированный ввод адресов** — исключает ошибки геокодинга, парсер существующих строк позволяет миграцию без потери данных
11. **Симметричные поля печати в шаблоне** — двусторонняя печать совпадает 1-в-1 при разрезе
12. **PDF через LibreOffice headless** — стандартное практическое решение; на сервере ~400 MB, на десктопе одной установкой
